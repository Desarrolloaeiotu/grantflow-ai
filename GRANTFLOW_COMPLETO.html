Integración Apollo.io

¿Qué es Apollo.io?

Plataforma SaaS de verificación y enriquecimiento de contactos. Valida emails, encuentra personas, obtiene información profesional.

3 Endpoints Implementados

POST /api/v1/contacts/verify

Verifica si un email es válido y obtiene info de contacto.

Request: { "email": "ceo@fundacion.org", "name": "John Doe" (opcional) } Response: { "verified": true, "confidence": "high", "email": "ceo@fundacion.org", "first_name": "John", "last_name": "Doe", "title": "Chief Executive Officer", "linkedin_url": "https://linkedin.com/in/johndoe", "company": "Fundación XYZ" }
POST /api/v1/contacts/enrich?funder_id={uuid}&limit=10

Busca contactos en una organización específica.

Response: { "status": "success", "funder_name": "LEGO Foundation", "contacts_found": 3, "contacts": [ { "email": "rasmus@legofoundation.org", "first_name": "Rasmus", "title": "President", "linkedin_url": "https://linkedin.com/in/rasmusnyrup" } ] }
POST /api/v1/opportunities/{id}/enrich-contacts

Enriquece una oportunidad completa con info de contactos.

Actualiza: ceo_email_verified, ceo_name, ceo_title, ceo_linkedin_url Response: { opportunity_id, status, enhancements: [], enhancements_count }

Patrón General n8n

1. Trigger (Schedule / Webhook / Manual) ↓ 2. Fetch Data (PostgreSQL query / HTTP request) ↓ 3. Transform (Filter, map, format) ↓ 4. Notify (Slack, Email, HTTP POST) ↓ 5. Log (Si error, alertar a #dev-alerts)

Futuro: Enrich-on-Scoring Workflow

Trigger: Cuando nueva opp es scored como GO ↓ Fetch opportunity desde DB ↓ IF decision == 'GO' THEN: - POST /opportunities/{id}/enrich-contacts (Apollo.io) - Slack notify: "✅ Nueva opp GO: [title]" - Email: CEO name, email, deadline ELSE: - Log: Opp no es GO, skip enriquecimiento

Preguntas Frecuentes

¿Por qué algunas oportunidades no tienen CEO verificado?

Apollo.io aún no está activado (se activa en Mes 5 en producción). Los endpoints funcionan con mock data en desarrollo. Una vez activado, verificará automáticamente todos los emails.

¿Qué significan los 5 criterios de scoring?

C1-C2 son evaluación cualitativa (LLM). C3-C5 son reglas cuantitativas (monto, tiempo, historial). Juntos determinan si una oportunidad es GO (≥6/10).

¿Se puede cambiar el scoring?

Sí. Los criterios se pueden ajustar en S7 (QA con equipo). Los prompts LLM están en `scoring_engine.py`. Las reglas cuantitativas están en las constantes.

¿Cuántas oportunidades nuevas se detectan diariamente?

Baseline: 20-50 oportunidades nuevas/día. Varía por fuente y estacionalidad de convocatorias.

¿Dónde exportar datos para el CRM?

En S6 se implementará endpoint GET /opportunities/export (CSV/XLSX). Por ahora: copiar/pegar desde grid o usar API directamente.

¿Qué pasa si un scraper falla?

El error se loguea, Slack es notificado (#dev-alerts), pero otros scrapers continúan. El sistema es resiliente a fallos individuales.

¿Cómo agregar una nueva fuente?

Crear nuevo scraper heredando de BaseScraper (ver `bid.py` como template). Implementar `fetch_raw()` y `normalize()`. Registrar en `runner.py`.

¿Cuánto cuesta mantener en producción?

Mes 1-4: ~$105-155/mes + Claude API variable. Mes 5+: +$49/mes Apollo.io. Mes 6+: +$299/mes Instrumentl (opcional).

¿Se puede integrarse con Salesforce/HubSpot?

Futuro (v2.0). Por ahora: exportar CSV e importar manualmente. En S8+ se planea sync automático.
